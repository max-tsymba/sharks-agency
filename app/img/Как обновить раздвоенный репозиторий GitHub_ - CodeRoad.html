В локальном клоне раздвоенного репозитория вы можете добавить исходный репозиторий GitHub как "remote". ("Remotes"-это как прозвища для URLs репозиториев - например, origin -это одно из них.) Затем вы можете извлечь все ветви из этого вышестоящего репозитория и перебазировать свою работу, чтобы продолжить работу над вышестоящей версией. С точки зрения команд, которые могут выглядеть так: 

# Add the remote, call it "upstream":

git remote add upstream https://github.com/whoever/whatever.git

# Fetch all the branches of that remote into remote-tracking branches,
# such as upstream/master:

git fetch upstream

# Make sure that you're on your master branch:

git checkout master

# Rewrite your master branch so that any commits of yours that
# aren't already in upstream/master are replayed on top of that
# other branch:

git rebase upstream/master

Если вы не хотите переписывать историю своей главной ветви (например, потому, что ее могли клонировать другие люди), то вам следует заменить последнюю команду на git merge upstream/master . Однако для того, чтобы сделать дальнейшие запросы на вытягивание максимально чистыми, вероятно, лучше перебазироваться.

Если вы перебазировали свою ветвь на upstream/master , вам может понадобиться принудительно нажать кнопку, чтобы переместить ее в свой собственный раздвоенный репозиторий на GitHub. Вы бы сделали это с помощью: 

git push -f origin master

Вам нужно использовать -f только в первый раз после того, как вы перебазировались.

Начиная с мая 2014 года, можно обновить fork непосредственно с GitHub. Это все еще работает по состоянию на сентябрь 2017 года, BUT это приведет к грязной истории фиксации(коммита).

  1. Откройте fork на GitHub.
  2. Нажмите на Pull Requests .
  3. Нажмите на New Pull Request . По умолчанию GitHub будет сравнивать оригинал с вашим fork, и там не должно быть ничего, чтобы сравнить, если вы не внесли никаких изменений.
  4. Нажмите кнопку switching the base , если вы видите эту ссылку. В противном случае вручную установите выпадающий список base fork на ваш fork, а head fork -на восходящий. Теперь GitHub будет сравнивать ваш fork с оригиналом, и вы должны увидеть все последние изменения.
  5. Create pull request и назначьте предсказуемое имя вашему запросу на вытягивание (например, Update from original ).
  6. Прокрутите вниз до Merge pull request, но пока ничего не нажимайте.

Теперь у вас есть три варианта, но каждый из них приведет к истории фиксации(коммита) less-than-clean.

  1. По умолчанию будет создан некрасивый коммит слияния.
  2. Если вы нажмете на выпадающий список и выберете "сквош и слияние", все промежуточные коммиты будут сжаты в один. Чаще всего это то, чего вы не хотите.
  3. Если вы нажмете Rebase and merge, все коммиты будут сделаны "with" вами, оригинал PRs будет связан с вашим PR, а GitHub отобразит This branch is X commits ahead, Y commits behind <original fork> .

Так что да, вы можете обновлять свое РЕПО с его восходящим потоком, используя GitHub web UI, но это запятнает вашу историю фиксаций(коммитов). Вместо этого придерживайтесь командной строки -это легко.

Вот официальный документ GitHub о синхронизации a fork :

Синхронизация a fork

Установка

Прежде чем вы сможете синхронизировать, вам нужно добавить пульт дистанционного управления, который указывает на вышестоящий репозиторий. Возможно, вы сделали это, когда изначально раздвоились.

Совет: синхронизация fork обновляет только локальную копию репозитория; она не обновляет ваш репозиторий на GitHub. 

$ git remote -v
# List the current remotes
origin  https://github.com/user/repo.git (fetch)
origin  https://github.com/user/repo.git (push)

$ git remote add upstream https://github.com/otheruser/repo.git
# Set a new remote

$ git remote -v
# Verify new remote
origin    https://github.com/user/repo.git (fetch)
origin    https://github.com/user/repo.git (push)
upstream  https://github.com/otheruser/repo.git (fetch)
upstream  https://github.com/otheruser/repo.git (push)

Синхронизация

Существует два шага, необходимых для синхронизации вашего репозитория с вышестоящим потоком: сначала вы должны получить данные с удаленного сервера, а затем объединить нужную ветвь в локальную ветвь.

Очаровательный

Извлечение из удаленного репозитория приведет к появлению его ветвей и соответствующих им коммитов. Они хранятся в вашем локальном репозитории в специальных ветвях. 

$ git fetch upstream
# Grab the upstream remote's branches
remote: Counting objects: 75, done.
remote: Compressing objects: 100% (53/53), done.
remote: Total 62 (delta 27), reused 44 (delta 9)
Unpacking objects: 100% (62/62), done.
From https://github.com/otheruser/repo
 * [new branch]      master     -> upstream/master

Теперь у нас есть главная ветвь upstream, хранящаяся в локальной ветви upstream/master 

$ git branch -va
# List all local and remote-tracking branches
* master                  a422352 My local commit
  remotes/origin/HEAD     -> origin/master
  remotes/origin/master   a422352 My local commit
  remotes/upstream/master 5fdff0f Some upstream commit

Слияние

Теперь, когда мы извлекли вышестоящий репозиторий, мы хотим объединить его изменения в нашу локальную ветвь. Это приведет эту ветвь в синхронизацию с восходящим потоком, не теряя наших локальных изменений. 

$ git checkout master
# Check out our local master branch
Switched to branch 'master'

$ git merge upstream/master
# Merge upstream's master into our own
Updating a422352..5fdff0f
Fast-forward
 README                    |    9 -------
 README.md                 |    7 ++++++
 2 files changed, 7 insertions(+), 9 deletions(-)
 delete mode 100644 README
 create mode 100644 README.md

Если в вашем локальном ветке не было никаких уникальных коммитов, git вместо этого выполнит "быструю перемотку вперед": 

$ git merge upstream/master
Updating 34e91da..16c56ad
Fast-forward
 README.md                 |    5 +++--
 1 file changed, 3 insertions(+), 2 deletions(-)

Многие ответы заканчиваются тем, что ваш fork один коммит опережает Родительский репозиторий. Этот ответ суммирует найденные здесь шаги, которые переместят ваш fork в ту же фиксацию(коммит), что и родительский .

  1. Измените каталог на свой локальный репозиторий.

    • Переключитесь на главную ветвь, если вы не git checkout master

  2. Добавьте родителя в качестве удаленного репозитория, git remote add upstream <repo-location>

  3. Выпуск git fetch upstream

  4. Выпуск git rebase upstream/master

    • На этом этапе вы проверяете, что фиксирует то, что будет объединено, набрав git status

  5. Выпуск git push origin master

Дополнительные сведения об этих командах см. В шаге 3 .

Если, как и я, вы никогда ничего не поручаете непосредственно мастеру , что вы действительно должны делать, вы можете сделать следующее.

Из локального клона вашего fork создайте свой вышестоящий пульт дистанционного управления. Вам нужно сделать это только один раз: 

git remote add upstream https://github.com/whoever/whatever.git

Затем всякий раз, когда вы хотите догнать вышестоящую главную ветвь репозитория, вам нужно: 

git checkout master
git pull upstream master

Предполагая, что вы никогда ничего не совершали по отношению к самому мастеру, вы уже должны быть сделаны. Теперь вы можете подтолкнуть свой локальный мастер к исходному удаленному GitHub fork. Вы также можете перебазировать свою ветвь разработки на ваш теперь up-to-date локальный мастер.

После первоначальной настройки upstream и проверки master все, что вам нужно сделать, это выполнить следующую команду для синхронизации вашего master с upstream: git pull upstream master .

Предисловие: ваш fork - это "origin", а хранилище, из которого вы раздвоились, - это "upstream".

Предположим, что вы уже клонировали свой fork на свой компьютер с помощью такой команды: 

git clone git@github.com:your_name/project_name.git
cd project_name

Если это дано, то вам нужно продолжать в этом порядке:

  1. Добавьте "upstream" в свой клонированный репозиторий ("origin"): 

    git remote add upstream git@github.com:original_author/project_name.git

  2. Извлеките коммиты (и ветви) из "upstream": 

    git fetch upstream

  3. Переключитесь на ветвь "master" вашего fork ("origin"): 

    git checkout master

  4. Спрячьте изменения вашей ветви "master": 

    git stash

  5. Объедините изменения из ветви "master" ветви "upstream" в ветвь "master" ветви "origin": 

    git merge upstream/master

  6. Разрешите конфликты слияния, если таковые имеются, и зафиксируйте слияние 

    git commit -am "Merged from upstream"

  7. Внесите изменения в свой fork 

    git push

  8. Верните свои припрятанные изменения (если таковые имеются) 

    git stash pop

  9. С тобой покончено! Поздравляю!

  • Откройте https://github.com/<repo>/commits/<hash>, где РЕПО - это ваш fork, а hash -полный hash коммит, который вы можете найти в вышестоящем веб-интерфейсе. Например, я могу открыть https://github.com/max630/linux/commits/0aa0313f9d576affd7747cc3f179feb097d28990, который указывает на linux master как время написания.

  • Введите имя новой ветви и нажмите Enter

Затем вы можете извлечь эту ветвь в свой локальный клон, и вам не придется возвращать все эти данные обратно в GitHub, когда вы нажимаете правки поверх этой фиксации(коммита). Или используйте веб-интерфейс, чтобы изменить что-то в этой ветке.

Как это работает (это предположение, я не знаю, как именно это делает GitHub): форки совместно используют хранилище объектов и используют пространства имен для разделения ссылок пользователей. Таким образом, вы можете получить доступ ко всем коммитам через ваш fork, даже если они не существовали к моменту разветвления.

В дополнение к этому ответу я искал способ обновить все удаленные ветви моего клонированного РЕПО ( origin ) из восходящих ветвей за один раз. Вот как я это сделал.

Это предполагает, что вы уже настроили вышестоящий удаленный указатель на исходный репозиторий (откуда был разветвлен origin ) и синхронизировали его с git fetch upstream .

Затем запустите: 

for branch in $(git ls-remote --heads upstream|sed 's#^.*refs/heads/##'); do git push origin refs/remotes/upstream/$branch:refs/heads/$branch; done

Первая часть этой команды перечисляет все головки в вышестоящем удаленном РЕПО и удаляет префикс имени ветви SHA-1, за которым следует префикс имени ветви refs/heads/ .

Затем для каждой из этих ветвей он перемещает локальную копию восходящей ветви удаленного отслеживания (refs/remotes/upstream/<branch> на локальной стороне) непосредственно в удаленную ветвь на исходной стороне ( refs/heads/<branch> на удаленной стороне).

Любая из этих команд синхронизации ветвей может завершиться неудачей по одной из двух причин: либо вышестоящая ветвь была переписана, либо Вы перенесли коммиты на эту ветвь в свой fork. В первом случае, когда вы ничего не зафиксировали в ветке на вашем fork, безопасно нажать принудительно (добавьте переключатель -f ; т. е. git push -f в приведенной выше команде). В другом случае это нормально, так как ваша ветвь fork разошлась, и вы не можете ожидать, что команда синхронизации будет работать до тех пор, пока ваши коммиты не будут объединены обратно в восходящий поток .

Когда вы клонировали свой раздвоенный репозиторий, перейдите к каталогу, в котором находится ваш клон, и нескольким строкам в вашем Git Bash Terminal. 

$ cd project-name

$ git remote add upstream https://github.com/user-name/project-name.git
 # Adding the upstream -> the main repo with which you wanna sync

$ git remote -v # you will see the upstream here 

$ git checkout master # see if you are already on master branch

$ git fetch upstream

Я хотел бы добавить к ответу @krlmlr's .

Изначально раздвоенный репозиторий имеет одну ветвь с именем: master . Если вы работаете над новой функцией или исправлением, вы обычно создаете новую ветвь feature и вносите изменения.

Если вы хотите, чтобы разветвленный репозиторий был синхронизирован с родительским репозиторием, вы можете настроить конфигурационный файл (pull.yml ) для приложения Pull (в ветке feature) следующим образом: 

version: "1"
rules:
  - base: feature
    upstream: master
    mergeMethod: merge
  - base: master
    upstream: parent_repo:master
    mergeMethod: hardreset

Это сохраняет ветвь master разветвленного РЕПО up-to-date с родительским РЕПО. Он сохраняет ветвь feature разветвленного РЕПО обновленной через ветвь master разветвленного РЕПО, объединяя их. Это предполагает, что ветвь feature является ветвью по умолчанию, которая содержит конфигурационный файл.

Здесь в игру вступают два mergemethods , один- hardreset , который помогает принудительно синхронизировать изменения в ветви master раздвоенного РЕПО с родительским РЕПО, а другой- merge . Этот метод используется для объединения изменений, сделанных вами в ветке feature , и изменений, сделанных из-за принудительной синхронизации в ветке master . В случае конфликта слияния приложение pull позволит вам выбрать следующий курс действий во время запроса pull.

Вы можете прочитать о базовых и расширенных конфигурациях и различных mergemethods здесь .

Есть две основные вещи, связанные с тем, чтобы разветвленный репозиторий всегда обновлялся навсегда.

1. Создайте ветви из Мастера fork и внесите в них изменения .

Таким образом, когда ваш запрос на вытягивание будет принят, вы можете безопасно удалить ветвь, так как ваш внесенный код будет затем жить в вашем мастере вашего раздвоенного репозитория, когда вы обновите его с помощью восходящего потока. Благодаря этому ваш мастер всегда будет в чистом состоянии, чтобы создать новую ветвь, чтобы сделать еще одно изменение.

2. Создайте запланированное задание для мастера fork для автоматического обновления.

Это можно сделать с помощью cron . Вот пример кода, если вы делаете это в linux. 

$ crontab -e

поместите этот код на crontab file , чтобы выполнить задание ежечасно. 

0 * * * * sh ~/cron.sh

затем создайте файл сценария cron.sh и взаимодействие git с ssh-агентом и / или ожидайте , как показано ниже 

#!/bin/sh
WORKDIR=/path/to/your/dir   
REPOSITORY=<name of your repo>
MASTER="git@github.com:<username>/$REPOSITORY.git"   
UPSTREAM=git@github.com:<upstream>/<name of the repo>.git  

cd $WORKDIR && rm -rf $REPOSITORY
eval `ssh-agent` && expect ~/.ssh/agent && ssh-add -l
git clone $MASTER && cd $REPOSITORY && git checkout master
git remote add upstream $UPSTREAM && git fetch --prune upstream
if [ `git rev-list HEAD...upstream/master --count` -eq 0 ]
then
    echo "all the same, do nothing"
else
    echo "update exist, do rebase!"
    git reset --hard upstream/master
    git push origin master --force
fi
cd $WORKDIR && rm -rf $REPOSITORY
eval `ssh-agent -k`

Проверьте свой раздвоенный репозиторий. Время от времени он всегда будет показывать это уведомление:

Эта ветвь даже с <upstream>: master .
